# 2026-04-13 批量验证任务进度条设计

## 背景
当前“批量验证并获取状态码”已经支持真实执行，但用户点击后仍然只能等待最终结果，无法看到处理中进度。随着验证逻辑已切换为 Playwright 页面探测，单次批量验证时间更长，因此需要在前端增加一个真实的任务进度条。

## 目标
把现有批量验证动作改造成任务化执行，让资产列表页在不跳转页面的情况下展示真实任务进度，包括总数、已处理数、成功数、失败数和当前状态。

## 范围
### 本次包含
- 批量验证改成任务流
- 后端保存最小验证任务状态
- 前端资产列表页内联显示验证进度条
- 前端轮询任务状态接口
- 验证完成后自动刷新资产列表

### 本次不包含
- 新建独立任务管理页面
- WebSocket / SSE
- 失败详情页
- 通用任务框架的大规模抽象重构

## 方案选择
### 方案 B（采用）
将批量验证任务化，前端在资产列表页展示真实进度条，并轮询任务状态。

理由：
- 最符合“任务进度条”的语义
- 比前端假进度条真实
- 比跳转实时日志页更直观
- 与当前批量验证逐步演进方向一致

## 后端设计
新增一个最小验证任务模型，至少包含：
- `task_id`
- `task_type = asset_verify`
- `status = pending | running | completed | failed`
- `total`
- `processed`
- `success`
- `failed`
- `message`
- `created_at`
- `updated_at`

### 接口设计
1. 创建验证任务
   - 前端点击按钮后调用创建接口
   - 后端立即返回 `task_id`

2. 查询任务状态
   - 前端根据 `task_id` 轮询
   - 返回当前进度字段

### 执行流程
- 创建任务记录
- 启动批量验证
- 每处理一条资产更新进度
- 全部完成后标记 `completed`
- 整体异常时标记 `failed`

## 前端设计
资产列表页内联显示真实进度条。

### 展示位置
放在资产列表页批量操作区域下方。

### 展示内容
- 进度条
- 当前文案：`正在验证 X / Y`
- 统计文案：`成功 A，失败 B`
- 任务状态文案

### 交互流程
1. 点击 `批量验证并获取状态码`
2. 前端拿到 `task_id`
3. 页面显示进度条
4. 每秒轮询任务状态
5. 完成后进度条到 100%
6. 自动刷新资产列表并显示结果提示

## 数据流
1. 用户勾选资产
2. 点击验证按钮
3. 后端创建 `asset_verify` 任务并返回 `task_id`
4. 后端执行 Playwright 验证并更新任务状态
5. 前端轮询任务状态接口
6. 前端更新进度条
7. 完成后刷新资产列表并展示状态码

## 异常处理
- 单条资产失败：不影响整批继续，`failed + 1`
- 整体任务失败：前端显示失败状态
- 最小版本不要求跨页面恢复任务显示

## 关键改动文件
预计涉及：
- `backend/app/api/assets.py`
- `backend/app/api/jobs.py` 或新增任务接口文件
- 可能新增任务模型 / schema 文件
- `frontend/src/views/AssetsView.vue`
- `frontend/src/api/modules/assets.ts`
- `frontend/src/types/index.ts`
