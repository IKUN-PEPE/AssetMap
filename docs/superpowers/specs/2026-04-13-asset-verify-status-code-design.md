# 2026-04-13 资产批量验证状态码设计

## 背景
用户发现资产列表中的“批量验证资产”按钮点击后，状态码列没有任何变化。排查代码后确认，当前后端 `POST /api/v1/assets/verify-batch` 仅更新 `verified` 字段，并不会实际请求资产 URL，也不会回填 `status_code`。因此当前行为与用户直觉不一致。

本次目标是把现有“批量验证资产”改成真正的在线验证能力：对所选资产的 `normalized_url` 发起真实请求，并把状态码写回资产记录，让前端刷新列表后直接显示结果。

## 目标
复用现有批量验证接口，在不新增页面和不引入任务队列的前提下，实现“批量验证并获取状态码”的最小可用版本。

## 范围
### 本次包含
- 复用 `POST /api/v1/assets/verify-batch`
- 后端对所选资产逐条请求 `normalized_url`
- 成功时写回 `status_code`
- 同时更新 `verified = true`
- 单条失败不中断整批处理
- 返回成功/失败统计
- 前端按钮文案与提示文案同步调整
- 前端刷新资产列表后直接显示状态码

### 本次不包含
- 异步任务队列
- 重试策略
- 失败原因详情查看
- 新建独立验证页面
- 新增另一套验证接口

## 方案选择
### 方案 A（采用）
保留现有按钮和接口，但将其行为改为真正的在线验证，并回填状态码。

理由：
- 最符合用户已经确认的目标
- 前端改动最小
- 后端改动集中在现有接口
- 能快速让“验证”一词与真实行为保持一致

### 备选方案
- 拆成两个按钮（标记已验证 / 在线验证）：语义更清晰，但超出本次最小目标
- 保持现有文案只做轻微修饰：不能解决用户要状态码的核心问题

## 后端设计
当前 `backend/app/api/assets.py` 中的 `/verify-batch` 只执行：
- 查询资产
- 设置 `verified`
- 提交数据库

本次修改为：
1. 查询所选资产
2. 逐条读取 `normalized_url`
3. 发起 HTTP 请求
4. 根据结果更新：
   - `status_code`
   - `verified = true`
5. 返回统计结果：
   - `updated`
   - `success`
   - `failed`

### 处理原则
- 单条请求成功：写入 `status_code`
- 单条请求失败：该条失败，但不影响整批继续执行
- 单条失败时，该资产仍视为已经执行过验证
- 本次不要求保存详细失败原因

## 前端设计
前端保持现有调用链路，只做最小调整：

1. 按钮文案从：
   - `批量验证资产`
   改为：
   - `批量验证并获取状态码`

2. 调用方式不变：
   - `verifyAssets(selectedIds)`

3. 成功提示更准确：
   - `批量验证完成，成功 X 条，失败 Y 条`

4. 调用成功后继续刷新资产列表，让状态码列直接反映最新数据。

5. 接口整体失败时，仍显示：
   - `资产验证失败`

## 数据流
1. 用户在资产列表勾选若干资产
2. 点击“批量验证并获取状态码”
3. 前端调用 `/api/v1/assets/verify-batch`
4. 后端逐条请求 `normalized_url`
5. 后端写回 `status_code` 和 `verified`
6. 前端刷新资产列表
7. 用户在状态码列看到结果

## 异常处理
### 单条 URL 请求失败
- 不影响整批其他资产继续执行
- 该条状态码不更新或保持空值
- 该条仍视为已完成验证流程

### 整批执行完成
- 返回成功/失败数量
- 前端展示统计提示

### 接口级异常
- 前端提示 `资产验证失败`
- 不额外引入复杂回滚机制
