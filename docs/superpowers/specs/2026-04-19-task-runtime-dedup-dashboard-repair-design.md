# 设计规格：任务执行链路、取消语义、去重一致性与仪表盘 API 接入修复

## 1. 背景与目标

当前分支已经完成任务中心、CSV 导入、自动后处理和 SOC 仪表盘的大部分功能，但存在四个会直接影响可用性的缺陷：

1. 任务启动链路使用进程内线程执行，但在线采集路径在子线程中调用事件循环的方式不安全，可能导致任务启动后立即失败。
2. “停止任务”当前只修改数据库状态，没有真正阻止后续采集与自动后处理，前后端语义不一致。
3. 重复 URL 的处理规则在不同导入链路中不一致，且仍会更新标题、状态码等字段，不符合“只更新最近发现时间”的要求。
4. 仪表盘前端直接写死 `http://127.0.0.1:9527/api/v1/stats`，导致环境变化或后端未使用该地址时页面无法显示数据。

本次修复目标是：在不重做现有任务中心架构的前提下，统一任务执行语义、取消语义、重复 URL 行为和仪表盘 API 接入方式，使当前实现稳定可用。

## 2. 范围

### 2.1 本次包含
- 修复进程内后台任务执行链路中的异步调用方式。
- 让任务取消变成后端可观察、可停止的协作式取消。
- 统一重复 URL 的处理逻辑为“只更新 `last_seen_at`”。
- 将仪表盘统计请求改为走现有前端统一 API 访问方式，移除硬编码服务地址。
- 为上述行为补充必要的后端/前端测试或最小验证覆盖。

### 2.2 本次不包含
- 不切换到真正的外部 Huey worker / consumer 部署模式。
- 不重构实时日志系统。
- 不扩展统计接口结构，也不在本次内重算 `rate=78` 的业务含义。
- 不清理当前分支中所有运行产物和构建产物。

## 3. 修复策略概览

采用“最小风险一致性修复”策略：

- 保留当前的进程内后台执行模型，不引入新的部署要求。
- 将 `run_in_process()` 明确视为“进程内后台执行器”，围绕它修正任务状态流转和取消检查点。
- 保持现有 API 结构和前端页面结构基本不变，只修复行为与接线问题。
- 通过统一的去重规则和取消检查，消除不同链路间行为漂移。

## 4. 任务执行链路设计

### 4.1 执行模型
后端继续使用进程内线程来启动 `run_collect_task`，但需要把该模型的语义固定下来：

- API 层负责校验任务是否存在、是否已在运行，并把任务状态切换到 `running`。
- `run_in_process()` 只负责在后台线程中调用任务函数，不承担队列调度语义。
- `run_collect_task` 作为唯一的采集入口，负责串联在线采集、CSV 导入和自动后处理。

这样做的目的是避免当前“既像 Huey 任务又像本地线程任务”的混合语义继续扩散。

### 4.2 线程中的异步执行
在线采集路径中，collector 的 `run()` 是异步方法。当前在线程中直接调用 `asyncio.get_event_loop()` 存在失败风险。

修复后应使用线程安全的异步执行方式：

- 在后台线程中使用独立的事件循环执行 collector。
- 不依赖线程默认事件循环是否存在。
- 每个 query 的异步执行应具备明确的边界：输入为 `query_str/query_payload/config`，输出为资产记录列表。

该改动只影响任务内部执行方式，不改变 API 入参、任务模型或 collector 对外接口。

## 5. 取消语义设计

### 5.1 取消模型
本次采用**协作式取消**，不实现线程级强杀：

- 用户调用停止接口后，任务状态被设置为 `cancelled`。
- 后台任务在关键边界主动检查数据库中的最新状态。
- 一旦检测到 `cancelled`，立即停止后续步骤并退出当前任务流。

### 5.2 取消检查点
取消检查至少覆盖以下阶段：

1. 任务开始执行前。
2. 每个 source/query 执行前。
3. 每个 source/query 执行后、写入数据库前。
4. 自动后处理（截图/验证）启动前。
5. 自动后处理内部迭代时的关键边界。

### 5.3 状态流转
任务状态保持以下语义：

- `pending`：已创建但尚未执行。
- `running`：后台执行中。
- `success`：主流程成功完成。
- `failed`：发生异常且未完成。
- `cancelled`：用户主动取消。

约束如下：

- 已取消的任务不能在结束时再被覆盖为 `success`。
- 已取消的任务不应继续触发自动后处理。
- 若执行过程中发生异常且状态尚未被取消，则置为 `failed` 并写入 `error_message`。

## 6. 重复 URL 规则统一设计

### 6.1 唯一性定义
重复资产继续通过 `normalized_url_hash` 识别。这与现有代码保持一致，避免扩大本次修复范围。

### 6.2 统一行为
命中已有 `WebEndpoint` 时，统一执行以下规则：

- 保留现有记录的 `title`、`status_code`、`source_meta`、`domain` 等业务字段。
- 不覆盖已有截图、验证状态或其他派生数据。
- 仅更新 `last_seen_at` 为本次观测时间。
- 如历史数据缺失 `first_seen_at`，允许仅在字段为空时补齐首见时间。

### 6.3 覆盖入口
以下入口必须统一：

- `backend/app/tasks/collect.py` 中的任务采集入库链路。
- `backend/app/services/collectors/import_service.py` 中的样例/CSV 导入链路。

### 6.4 去重策略兼容
`CollectJob.dedup_strategy` 字段保留，以避免影响现有 UI 和接口，但本次修复将其行为收敛：

- `skip`：发现重复记录时，不新建资产，记录重复数，并按统一规则刷新 `last_seen_at`。
- `overwrite`：本次不再覆盖已有资产业务字段；行为与 `skip` 对齐，仅保留统计与兼容字段语义。

之所以这样设计，是因为用户已经明确要求“重复 URL 只更新最近发现时间”，该规则优先于之前的字段覆盖能力。

## 7. 仪表盘 API 接入设计

### 7.1 前端访问方式
`DashboardView.vue` 不再直接使用硬编码的绝对地址访问统计接口，而是改为走现有前端统一 API 基础设施：

- 使用项目内统一 HTTP 客户端或模块封装。
- 请求路径改为相对路径 `/api/v1/stats/...`。
- 由当前前端环境、代理或部署配置决定最终目标地址。

### 7.2 页面行为
保留现有页面结构：

- KPI 卡片
- 来源分布图
- 验证状态图
- 趋势图

保留加载失败时的降级显示，但不能再因为硬编码地址导致所有环境下的页面默认失效。

### 7.3 后端兼容性
`backend/app/api/statistics.py` 的接口结构保持不变：

- `GET /overview`
- `GET /distribution`
- `GET /trends`

本次只修接入方式，不扩展返回结构。

## 8. 组件边界与职责

### 8.1 `backend/app/api/jobs.py`
负责：
- 接收启动/停止请求。
- 执行任务状态的入口级校验。
- 调用后台执行器。

不负责：
- 实际采集逻辑。
- 线程中断。
- 去重细节。

### 8.2 `backend/app/core/huey.py`
负责：
- 提供进程内后台执行能力。

不负责：
- 任务状态管理。
- 取消判定。
- 业务重试语义。

### 8.3 `backend/app/tasks/collect.py`
负责：
- 任务主流程。
- 在线采集与 CSV 导入分支。
- 取消检查。
- 任务统计更新。
- 自动后处理入口控制。

### 8.4 `backend/app/services/collectors/import_service.py`
负责：
- 单条记录导入时的资产/服务/观测写入。
- 与任务链路一致的重复 URL 更新规则。

### 8.5 `frontend/src/views/DashboardView.vue`
负责：
- 展示统计数据。
- 调用统一 API 接入层。
- 在请求失败时保持页面可用。

## 9. 错误处理设计

### 9.1 任务执行异常
- collector 执行失败时，记录 `error_message`。
- 若任务未被取消，则置为 `failed`。
- 已完成计数不回滚为 0，保留已成功处理的部分结果。

### 9.2 取消优先级
- 若用户已取消任务，最终状态应保留 `cancelled`。
- 即便后续流程某一步报错，也不应把已取消任务重新写成 `failed` 或 `success`，除非取消前就已发生不可恢复异常并完成状态写入。

### 9.3 仪表盘请求失败
- 前端显示默认/空态数据。
- 在控制台保留错误日志，便于调试。
- 页面结构不应因请求失败而消失。

## 10. 测试与验证

### 10.1 后端验证重点
1. 在线采集任务在后台线程中可正常执行异步 collector。
2. 停止任务后，主流程会在下一检查点终止，且状态最终为 `cancelled`。
3. 自动后处理不会在已取消任务上继续运行。
4. 重复 URL 在任务链路和导入链路上都只更新 `last_seen_at`。

### 10.2 前端验证重点
1. 仪表盘请求走相对 API 路径，不再依赖固定端口地址。
2. 后端可用时，统计数据正常显示。
3. 后端不可用时，页面仍保持完整框架和降级数据。

### 10.3 回归关注点
- 任务列表页面仍可创建、启动、停止并轮询状态。
- CSV 预览与字段映射流程不受影响。
- 自动验证开关仍按现有 UI 行为工作。

## 11. 实施顺序建议

1. 修复后台线程中的异步 collector 执行方式。
2. 为任务主流程补齐取消检查与状态保护。
3. 统一 `collect.py` 与 `import_service.py` 的重复 URL 行为。
4. 调整 `DashboardView.vue` 的统计接口接入方式。
5. 补充验证并进行回归测试。

## 12. 预期结果

修复完成后，应达到以下结果：

- 在线采集任务不会因线程事件循环问题而随机失败。
- 停止任务后，后端会实际停止后续处理，而不是只修改界面状态。
- 重复 URL 在所有导入链路中只更新最近发现时间。
- 仪表盘与当前运行环境解耦，不再依赖写死的本地服务地址。
