# 2026-04-13 实时日志页面设计

## 背景
用户希望在前端新增一个独立的“实时日志”页面，用于查看任务日志和后端服务日志。当前项目前端已有管理后台布局与路由体系，后端已有任务、截图、资产等接口，但没有现成的实时日志展示页，也没有面向前端的统一日志读取接口。

本次目标是先交付一个最小可用版本，优先保证可落地、结构清晰、后续可扩展。用户已明确选择：
- 使用独立日志页面
- 同时查看任务日志和服务日志
- 实时方式先采用轮询
- 展示风格为“结构化列表 + 控制台视觉风格”

## 目标
新增一个独立的“实时日志”页面，接入后台导航，通过前端轮询后端统一日志接口的方式，实现对任务日志、服务日志、全部日志的查看与切换。

## 范围
### 本次包含
- 新增独立日志页面
- 新增前端路由与导航入口
- 新增后端统一日志读取接口
- 支持日志来源切换：任务日志 / 服务日志 / 全部
- 支持自动轮询刷新
- 支持暂停 / 继续刷新
- 支持清空当前前端视图
- 支持空状态与错误状态提示
- 支持控制台风格展示结构化日志

### 本次不包含
- WebSocket / SSE
- 复杂全文搜索
- 日志下载/导出
- 日志持久化管理后台
- 多用户日志订阅权限体系

## 方案选择
### 方案 A（采用）
独立日志页 + 前端轮询 + 后端统一日志接口。

理由：
- 最符合用户确认的实现方向
- 适合当前项目现状，改动成本低
- 不需要先引入流式通信基础设施
- 未来可平滑升级到 SSE

### 备选方案
- SSE：实时性更强，但当前项目缺少基础设施，超出本次最小范围
- WebSocket：扩展性最好，但明显偏重，不适合当前目标

## 页面设计
### 1. 顶部工具栏
包含：
- 日志来源切换：任务日志 / 服务日志 / 全部
- 自动刷新状态展示
- 暂停按钮
- 继续按钮
- 清空视图按钮

### 2. 主日志区域
视觉上采用深色控制台风格。

每条日志包含结构化字段：
- 时间 `timestamp`
- 级别 `level`
- 来源 `source`
- 消息 `message`

展示特点：
- 按日志级别上色
- 默认展示最新日志
- 默认自动滚动到底部
- 用户手动上滚时暂停自动贴底

### 3. 底部状态栏
展示：
- 当前显示条数
- 最近更新时间
- 当前轮询间隔

## 数据流设计
### 后端接口
新增统一日志读取接口，支持参数：
- `source`: `task` / `service` / `all`
- `limit`: 最近返回条数
- `since`: 增量拉取时间点

返回结构建议：

```json
{
  "items": [
    {
      "timestamp": "2026-04-13T12:00:00Z",
      "level": "info",
      "source": "task",
      "message": "截图任务开始执行"
    }
  ],
  "next_since": "2026-04-13T12:00:00Z"
}
```

### 前端轮询
- 页面首次进入：拉最近一批日志
- 之后每隔 1~2 秒拉取增量日志
- 暂停时停止轮询
- 恢复时继续按最新时间点增量拉取
- 本地仅保留最近 200~500 条日志

## 日志来源策略
### 任务日志
优先覆盖：
- 导入任务
- 截图任务
- 标签相关任务

### 服务日志
接入后端服务运行日志。

### 降级原则
如果某类日志当前项目中没有完整来源，可以先返回基础日志数据，但接口结构必须稳定，避免影响前端页面落地。

## 交互与异常处理
### 接口失败
- 页面提示“日志拉取失败”
- 保留上一次成功日志内容
- 后续轮询自动继续重试

### 无日志数据
按来源显示空状态：
- 当前暂无任务日志
- 当前暂无服务日志
- 当前暂无日志

### 日志过多
- 前端只保留最近固定条数
- 避免页面卡顿

### 清空视图
- 只清空前端当前展示
- 不删除后端日志
